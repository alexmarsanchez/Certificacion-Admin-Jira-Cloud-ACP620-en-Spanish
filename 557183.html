Ramas de automatización de Jira

Al configurar reglas de automatización, es posible crear una sección independiente de la regla y realizar acciones sobre problemas relacionados; esto se conoce como ramificación . A continuación, encontrará los tipos de ramas disponibles en Jira Cloud Automation. Para obtener más información, consulte ¿Qué es la ramificación de reglas ?

Cuestiones relacionadas

Las ramas de incidencias relacionadas permiten realizar acciones sobre incidencias relacionadas con la incidencia que activó la regla. Por ejemplo, se puede tener una regla que se activa cuando una incidencia pasa a Listo, donde se agrega un comentario en las subtareas de esa incidencia.

Acceder a los problemas creados

Las reglas pueden crear incidencias mediante las acciones "Crear incidencia" y "Clonar incidencia" . Para realizar otras acciones, como añadir un comentario o crear subtareas, en estas incidencias recién creadas dentro de la misma regla, se requiere una rama de incidencia relacionada.

Esto se debe a que la rama principal de una regla siempre se aplica al problema desencadenante, no al problema creado. Por ejemplo, añadir un comentario sobre el problema después de crearlo , añade un comentario al problema desencadenante, no al problema creado.

Para solucionar esto, cree una nueva rama para Todos los problemas creados para permitirle actuar sobre los problemas recién creados.

Alternativamente, puede utilizar la condición Problemas relacionados ( Creados más recientemente ) si solo necesita solucionar un solo problema.

Orden de ejecución de ramas

  • Las ramas de varias incidencias (como 4 subtareas) se ejecutarán en paralelo sin garantía de que una finalice antes del inicio de la siguiente. Por lo tanto, no se puede confiar en los cambios entre ramas.

  • Las ramas de varios problemas se ejecutan como un nuevo proceso y la rama principal continúa su ejecución antes de que comience la rama secundaria.

Ramificación avanzada

La ramificación avanzada permite introducir un valor inteligente y ejecutar acciones sobre él como un objeto. Por ejemplo, podría introducir el valor inteligente {{triggerissue.comments.body}}, lo que permitiría que la rama ejecutara acciones para cada comentario del problema.

Desencadenantes de automatización de Jira

Cada regla comienza con un disparador. Este inicia la ejecución de tus reglas. Los disparadores detectan eventos en Jira, como la creación de una incidencia o la modificación del valor de un campo.

Aplique condiciones directamente al disparador para añadir criterios a toda la regla. Esto filtrará los disparadores que continúan con la automatización.

Los activadores se pueden configurar para que se ejecuten según un cronograma y se pueden personalizar antes de aplicarlos a una regla.

Vea cómo usar todos estos activadores en nuestra biblioteca de plantillas de automatización de Jira.

Desencadenantes generales

Estos activadores se pueden utilizar en todos los productos de Jira Cloud.

El valor del campo ha cambiado

  • Valores inteligentes relacionados : {{fieldChange}}

La regla se ejecutará al cambiar el valor de un campo. Este disparador es compatible con todos los campos del sistema y personalizados.

Puedes usar este disparador junto con condiciones para comprobar el valor de los campos antes de realizar acciones. Por ejemplo, envía un SMS cuando la prioridad de un problema cambie a mayor que alta.

Al configurar esta regla, puede seleccionar los campos que desea supervisar en busca de cambios o usar una expresión regular para que coincidan con los nombres de los campos. Puede limitar los campos a supervisar especificando un tipo de cambio: valor añadido, valor eliminado o cualquier cambio en el valor del campo. También puede limitar las operaciones de incidencia que activarán esta regla (crear, editar, transferir o asignar) o dejarla en blanco para supervisar todas las operaciones.

Formulario adjunto

Los activadores de formulario son solo específicos del proyecto y no se pueden configurar de forma global.

La regla se ejecutará siempre que se adjunte manualmente un formulario a un problema existente, pero también cuando se adjunte un formulario mediante la API Agregar formulario o se copie un formulario a otro problema mediante la API Copiar formulario .

Formulario abierto para ediciones

Los activadores de formulario son solo específicos del proyecto y no se pueden configurar de forma global.

La regla se ejecutará siempre que se abra un formulario para que el cliente edite elementos de trabajo existentes. Por ejemplo, puede haber ocasiones en las que un agente seleccione " Abrir para editar" en un formulario ya enviado para un elemento de trabajo. Esto podría ser el disparador que envía una notificación al cliente para que vuelva a enviar el formulario. Este disparador se ejecutará siempre que se abra un formulario manualmente para editarlo, pero también cuando se abra mediante la API "Reabrir formulario" .

Formulario enviado

Los activadores de formulario son solo específicos del proyecto y no se pueden configurar de forma global.

La regla se ejecutará al enviar los formularios adjuntos a un problema. Al configurar el disparador, puede seleccionar cualquier número de formularios, lo que cambiará su funcionamiento:

  • Si no se selecciona ningún formulario : la regla se ejecutará cuando se envíe cualquier formulario.

  • Si se selecciona un formulario : la regla se ejecutará cuando se envíe ese formulario.

  • Si se seleccionan varios formularios : la regla solo se ejecutará cuando se envíen todos los formularios seleccionados en un problema.

Webhook entrante

  • Valores inteligentes relacionados : {{webhookData}}

La regla se ejecutará cuando  HTTP POST se envíe a una URL de webhook específica.

Un webhook permite a un tercero activar una regla de automatización. El webhook puede especificar problemas sobre los que actuar o incluso proporcionar datos en tiempo real que permiten actualizar un problema.

Al configurar este disparador, recibirá una URL y un secreto únicos que puede agregar a la configuración del webhook saliente de la aplicación de terceros o realizar una  HTTP POST solicitud desde sus scripts personalizados. El disparador también proporciona detalles sobre cómo proporcionar claves de emisión y otros datos.

Puede utilizar el  {{webhookData}} valor inteligente para hacer referencia a los datos XML personalizados proporcionados por el webhook en su regla.

Problema asignado

  • Valores inteligentes relacionados : {{assignee}}

La regla se ejecutará al cambiar el responsable de un problema. Por ejemplo, al asignar un problema a un usuario específico, cambie su estado a "  En curso"  y envíe un correo electrónico al informante para informarle que se está investigando.

Problema comentado

  • Valores inteligentes relacionados : {{comment}}

La regla se ejecutará al añadir un nuevo comentario. Por ejemplo, cambie el estado a "  En progreso"  al añadir un nuevo comentario a un problema.

Elegir un tipo de comentario permite distinguir si un comentario se añadió de forma independiente o como parte de una transición. Por ejemplo, puede configurar una regla de automatización para que un comentario añadido a una incidencia cerrada se reabra o para que un comentario añadido cambie el estado de la incidencia. Los tipos de comentario disponibles son:

  • Todos los comentarios (predeterminado): todos los comentarios que se agregan a un problema.

  • Comentar es la acción principal: el nuevo comentario se agrega como un mensaje en la sección de comentarios.

  • Comentario agregado durante la transición de estado: el comentario se agrega cuando se cambia el estado de un problema.

  • Comentario agregado mientras se editaban los campos del problema: Comentario agregado mientras se editaban uno o más campos del problema.

Editar un comentario no activará esta regla.

Comentario del problema editado

  • Valores inteligentes relacionados: {{comment}}

Esta regla se ejecutará al editar un comentario. Se puede usar con la  acción "Editar comentario" .

Problema creado

  • Valores inteligentes relacionados : {{issue}}

La regla se ejecutará cuando se cree un problema.

Puede utilizar este disparador con acciones para personalizar el nuevo problema, lo que incluye completar campos, asignarlos a usuarios y agregar subtareas.

Problema eliminado

  • Valores inteligentes relacionados : {{issue}}

La regla se ejecutará al eliminar un problema. Por ejemplo, se enviará una notificación por correo electrónico indicando que se ha eliminado un problema.

Puede utilizar condiciones para refinar exactamente el problema que está monitoreando.

Problema relacionado

  • Valores inteligentes relacionados :  {{destinationIssue}}, {{linkType}}

La regla se ejecutará cuando un problema esté vinculado a otro problema.

Puede configurar este disparador en función de cualquier tipo de enlace.

Enlace del problema eliminado

  • Valores inteligentes relacionados :  {{destinationIssue}}, {{linkType}}

La regla se ejecutará cuando un problema no esté vinculado a otro.

Puede configurar el disparador para que se ejecute solo para tipos de enlaces específicos o para todos los enlaces de problemas.

Problema trasladado

  • Valores inteligentes relacionados : {{issue}}

La regla se ejecutará cuando un problema se traslade de un proyecto a otro.

Puede utilizar este disparador junto con condiciones y acciones para garantizar que todos los valores, campos y configuraciones se copien al nuevo proyecto.

Problema en transición

  • Valores inteligentes relacionados :  {{issue}}, {{changelog}}

La regla se ejecutará cuando un problema pase de un estado a otro.

Puede configurar este disparador para que escuche el estado que elija o simplemente cualquier transición en su flujo de trabajo.

Obtenga más información sobre cómo migrar un problema con automatización

Número actualizado

  • Valores inteligentes relacionados : {{issue}}

La regla se ejecutará cuando se actualicen los detalles de un problema.

Las excepciones de este disparador incluyen cambios realizados por las  acciones Vincular problema ,  Asignar problema y  Registrar trabajo  .

Para activar una regla cuando se actualiza el estado de un problema, utilice en su lugar el activador Problema en transición .

Disparador manual por problema

  • Valores inteligentes relacionados : {{issue}}

La regla se ejecutará cuando un usuario la active manualmente. Este activador es útil para automatizar tareas comunes o para probar o depurar una regla. Obtenga más información sobre cómo usar el activador manual para probar reglas.

Puede definir qué usuarios pueden activar manualmente una regla seleccionando un grupo de permisos en el menú desplegable "Grupos que pueden ejecutar el activador" . Más información sobre los grupos de permisos.

También puede solicitar información antes de ejecutar una regla seleccionando " Obtener información de los usuarios" . Esto le permite configurar algunos campos que se mostrarán a los usuarios cuando activen la regla. Puede configurar cada campo como obligatorio seleccionando " Campo obligatorio" . Si un usuario decide no introducir información y selecciona "Cancelar" , no se contabilizará como ejecución para sus límites mensuales.

Una vez que se crea una regla con un disparador manual, cualquier persona con acceso podrá activarla yendo a un problema y seleccionando Acciones .

Eventos de emisión múltiple

  • Valores inteligentes relacionados : {{issue}}

La regla se ejecutará cuando se produzcan uno o más eventos de incidencia seleccionados. Por ejemplo, enviar un mensaje de Slack cuando se actualice, transfiera o asigne una incidencia.

Usar este disparador puede ser más fácil y eficiente que crear varias reglas diferentes.

Proyecto creado

  • Valores inteligentes relacionados: Valores inteligentes de automatización - proyectos

La regla se ejecutará al crear un proyecto. Por ejemplo, esto podría ser útil si desea crear un conjunto de incidencias predeterminadas que se incluirán en todos los proyectos de Jira.

Programado

  • Valores inteligentes relacionados : {{issue}}

Esta regla se ejecuta según una programación específica. Puede ejecutarla con una frecuencia fija (por ejemplo, cada 7 días) o usar una  expresión Cron  para programaciones más complejas.

También puede optar por introducir una consulta JQL. Si lo hace, las acciones de esta regla se ejecutarán en los problemas incluidos en la consulta.

Las reglas programadas que alcancen un estado de Falla durante 10 ejecuciones consecutivas se deshabilitarán automáticamente.

Trabajo registrado

  • Valores inteligentes relacionados : {{worklog}}

Esta regla se ejecutará cuando se cree, actualice y/o elimine un registro de trabajo.

Condiciones de automatización de Jira

Puedes especificar las condiciones que deben cumplirse para que tu regla siga ejecutándose. Por ejemplo, solo escalará un problema si es de alta prioridad.

Las condiciones pueden colocarse en cualquier punto de la cadena de reglas. Si una condición falla, no se realizará ninguna acción posterior. La excepción es la condición del bloque "If/else" . No todas las reglas necesitan condiciones.

La ubicación de las condiciones en una regla afectará su funcionamiento.

Asignar problema

  • Utilice valores inteligentes aquí : Sí

  • Permisos necesarios : Asignar problemas, Usuario asignable

Asigna un problema a un usuario.

Hay ocho opciones diferentes para asignar problemas:

  • Un usuario en una lista definida : Una incidencia que supera una restricción (especificada en JQL) se asigna a un usuario de una lista específica de usuarios mediante el método que usted elija. Esto puede ser:

    • Carga de trabajo equilibrada : garantiza que cada usuario de la lista tenga asignado el mismo número de problemas. El alcance de estos problemas se define mediante el JQL.

    • Aleatorio : se asigna aleatoriamente de la lista de usuarios.

    • Round-Robin : asigna problemas según la secuencia en la lista de usuarios especificados.

  • Otro valor de campo o comentario : Asigne la incidencia según los valores de los campos de usuario de la incidencia o de un comentario. Por ejemplo, asigne la incidencia a un usuario anterior según su historial o a su creador. Si especifica usuarios para excluir o grupos a los que debe pertenecer, se omitirá cualquier usuario que no cumpla este requisito y la acción continuará buscando usuarios coincidentes (por ejemplo, buscando en el historial de incidencias de los usuarios asignados).

  • Automático : asigna el problema al asignado predeterminado del proyecto.

  • Copiar de otra incidencia : Copiar al asignado de una incidencia relacionada. También puedes usar JQL para encontrar una incidencia de la que copiar al asignado.

  • Valor inteligente : Asigna el problema según un valor inteligente. Por ejemplo, cuando se comenta un problema, puedes asignarlo al autor del comentario. Más información sobre los valores inteligentes de usuario.

  • Especificar usuario : Asignar a un usuario seleccionado.

  • Sin asignar : elimine el usuario existente y establezca el problema como "Sin asignar".

  • Usuario que activó el evento : asigna el problema al usuario que activó el evento.

  • Usuario en un rol: asigna el problema a un rol específico dentro de un equipo.

  • Usuario en un grupo: asigna el problema a un usuario de un grupo seleccionado.

  • Respondedor de guardia : Asigne el problema a quien esté de guardia, según el horario de guardia de Opsgenie del equipo. Los respondedores se asignan aleatoriamente si el horario seleccionado incluye varias rotaciones.

Valores ocultos

Al configurar su solicitud web, también puede ocultar ciertos valores para aumentar su seguridad. Si un valor se marca como oculto y se guarda la regla, este se reemplazará por asteriscos (por ejemplo, *****); dejará de ser visible y no podrá inspeccionar la carga útil. Una vez que un valor se oculta, no se puede revertir (es decir, la casilla " Oculto" no se puede desmarcar), aunque el valor sí se puede cambiar al editar la regla.

Por razones de seguridad, los valores ocultos se perderán si se duplica o exporta/importa toda la regla, o si se duplica el componente "Enviar solicitud web". En este caso, deberá reconfigurar el componente y volver a introducir esos valores.

Otras cosas a tener en cuenta

  • Puertos permitidos: tenga en cuenta que los únicos puertos permitidos para las URL de la acción Enviar solicitud web son 80, 8080, 443, 6017, 8443, 8444, 7990, 8090, 8085, 8060, 8900, 9900.

  • Sensibilidad de los datos : La acción "Enviar solicitud web" se puede configurar para proteger la información confidencial de terceros. Esto se aplica incluso después de guardar la regla, ya que alguien con permiso para editarla puede reconfigurarla para enviar datos que no debería. Por este motivo, recomendamos que, antes de usar esta acción, se asegure de que solo personas de confianza puedan editar las reglas de automatización.

Retrasar hasta que se produzca un evento de alerta

Esta acción está disponible solo para todos los planes Premium y Enterprise.

  • Utilice valores inteligentes aquí : No

La acción de retraso basada en eventos permite retrasar la ejecución de una regla hasta 15 minutos hasta que se produzca un evento específico en una entidad de alerta relacionada. Los valores inteligentes de la alerta se recuperarán antes de iniciar el retraso y después de recibir el evento. Esta función es especialmente útil en la gestión de incidentes, donde es crucial validar el cumplimiento de ciertas condiciones o métricas antes de continuar con el siguiente paso de una regla.

Eventos admitidos:

  • Prioridad cambiada

  • Estado actualizado

Configuración:

  • Los administradores pueden elegir qué campo (por ejemplo, Prioridad o Estado) supervisar para detectar cambios. La regla se pausará hasta 15 minutos hasta que se produzca el evento especificado en este campo.

  • Define la condición que debe cumplirse para que el evento active la continuación de la regla. Las condiciones disponibles incluyen:

    • Igual

    • No es igual

    • Es uno de

    • No es uno de

Seleccionar valor:

Los administradores también deben especificar los valores que debe cumplir el campo de evento para activar la continuación de la regla.

Ejecución de la regla:

  • La regla se detiene hasta el tiempo de retraso establecido o hasta que ocurre el evento especificado, lo que ocurra primero.

  • Si el evento no ocurre dentro del tiempo de retraso establecido, la regla se reanuda automáticamente, con una ligera latencia en casos excepcionales.

  • La regla no abortará la ejecución si el evento no ocurre; continuará después del retraso.
